Projeto de Mentoria — Processamento de CSV com Spring Boot

Objetivo do Projeto

Construir uma aplicação Spring Boot capaz de:

Receber um CSV (upload ou link)

Validar o conteúdo

Enviar o arquivo para Cloud Storage

Processar os jogos assincronamente enriquecendo com a API RAWG

Expor endpoint de status

(Opcional) Proteger com autenticação e autorização


A evolução será incremental, sempre garantindo algo funcional ao final de cada etapa.

Decisão Arquitetural Importante

O StorageService fica em shared/storage porque:

É uma abstração reutilizável

Não pertence a um domínio específico

Permite trocar implementação (S3, GCS etc.)


As implementações concretas ficam em:

shared/storage/impl

Exemplo:

S3StorageService

GcsStorageService



---

O RawgClient fica dentro de:

games/integration

Porque é uma integração específica do contexto de jogos.

FASE 2 — Upload e Validação de CSV

Objetivo

Receber um MultipartFile e validar estrutura mínima do CSV.


---

Onde implementar

Inicialmente dentro de:

fileprocessing/service

Se crescer muito, extrair para:

fileprocessing/service/CsvParserService.java


---

Fluxo

1. Receber arquivo via MultipartFile


2. Validar se possui coluna obrigatória (ex: game_name)


3. Se inválido → lançar exceção customizada


4. Se válido → criar ProcessingRequest com status:



RECEIVED

Ainda não é necessário enviar para cloud nessa etapa.


---

O que pesquisar

MultipartFile

Leitura de CSV (OpenCSV ou BufferedReader)

Streams em Java

Tratamento global de exceção com @ControllerAdvice

Criação de exceções customizadas



---

Objetivo pedagógico

Aprender:

Manipulação de arquivos

Validação de dados

Tratamento de erro

Organização de responsabilidades



---

FASE 3 — Cloud Storage

Objetivo

Enviar o CSV validado para um bucket de armazenamento.


---

Onde implementar

Interface:

shared/storage/StorageService.java

Responsabilidades da interface:

upload(File file)

generateDownloadUrl(String path)



---

Implementação concreta

shared/storage/impl/S3StorageService.java

ou

shared/storage/impl/GcsStorageService.java


---

Configuração

config/StorageConfig.java


---

Fluxo

O service de fileprocessing deve chamar a interface:

StorageService

Nunca depender da implementação concreta.


---

O que pesquisar

Injeção de dependência

@Configuration

@Bean

Variáveis de ambiente

SDK do provedor (S3 ou GCS)

Conceito de abstração e desacoplamento



---

Critério para avançar

Se trocar S3 por GCS, apenas a implementação muda.

O service não deve conhecer detalhes do provider.

FASE 4 — Processamento Assíncrono

Objetivo

Processar os jogos em background após o upload.


---

Onde implementar

Classe de job:

fileprocessing/job/ProcessingJob.java

Configuração:

config/AsyncConfig.java


---

Responsabilidades do Service

O service deve:

1. Marcar status como:



PROCESSING

2. Disparar método anotado com:



@Async

3. Após finalizar:



DONE se sucesso

ERROR se falha



---

O que pesquisar

@EnableAsync

@Async

CompletableFuture

Noções básicas de concorrência

Transações em métodos assíncronos



---

Objetivo pedagógico

Introduzir:

Assincronismo

Separação entre requisição HTTP e processamento real
